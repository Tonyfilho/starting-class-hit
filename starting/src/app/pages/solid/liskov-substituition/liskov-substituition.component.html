Liskov Substitution Principle (LSP)

O que é o Liskov Substitution Principle?

O LSP afirma que as subclasses devem poder ser usadas no lugar das superclasses sem alterar o comportamento esperado.
Isso mantém a consistência e a previsibilidade do sistema.

Exemplo violando o LSP

Subclasse quebra o contrato


    class Ave {
        voar() { console.log("Voando"); }
    }

    class Pinguim extends Ave {
        voar() { throw new Error("Pinguins não voam"); }
    }

A substituição falha: o pinguim quebra o comportamento esperado da classe Ave.

Aplicando o LSP

Separando comportamentos


    interface Ave { andar(): void; }
    interface AveQueVoa extends Ave { voar(): void; }

    class Andorinha implements AveQueVoa {
        andar() { ... }
        voar() { ... }
    }

    class Pinguim implements Ave {
        andar() { ... }
    }

Agora cada ave implementa apenas o que realmente faz.

Resumo Técnico

  • ✅ Substituições devem ser seguras e previsíveis.
  • ✅ Evita comportamentos inesperados ou erros em tempo de execução.
  • ✅ Promove o design baseado em contratos claros.
