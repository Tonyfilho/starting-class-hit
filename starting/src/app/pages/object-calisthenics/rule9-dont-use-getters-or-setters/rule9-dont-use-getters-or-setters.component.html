Rule 9 - Don’t Use Getters/Setters

Avoid accessing internal data from outside the object.

Why Avoid Getters and Setters?

In Object Calisthenics, exposing internal state through getters or modifying it with setters breaks encapsulation. It treats objects like data containers rather than behavior-driven entities.

Angular Context

In Angular components or services, instead of using a getter to expose data (like a user name), prefer exposing a method that expresses an action or behavior.

❌ Bad Example:

export class User {{"{"}}
  private name = 'Ana';

  getName() {{"{"}} return this.name; {{"}"}}
{{"}"}}

And then using it like:

const userName = user.getName();
console.log('Welcome, ' + userName);

✅ Good Example:

export class User {{"{"}}
  private name = 'Ana';

  greet(): string {{"{"}}
    return `Welcome, ${{"{"}}this.name{{"}"}}`;
  {{"}"}}
{{"}"}}

Then use it like:

console.log(user.greet());

When Using Getters Makes Sense

While Object Calisthenics recommends avoiding them, Angular heavily uses getters for performance or template bindings. The goal here is to **not abuse getters as shortcuts to avoid encapsulating logic**.

⚠️ Use getters sparingly in Angular, especially in templates with change detection, and never as a replacement for business logic.
